# Event Schema Validation

## Summary

- No `output/` directory with generated event files was present in the workspace.
- Validation is based on the provided sample file `sample_eventsbe42122.jsonl`, the pipeline event definition in `pipeline/events.py`, the generated output pattern in `pipeline/run.py`, and the API validation schema in `app/schemas/event.py`.
- The sample file schema is not aligned with the repository's expected event schema and would cause ingestion or submission failure.

## Expected Event Schema

The repository defines two relevant event models:

1. Pipeline generation model (`pipeline/events.py`):
   - `event_id`: UUID
   - `store_id`: int
   - `camera_id`: str
   - `visitor_id`: str
   - `event_type`: str
   - `timestamp`: ISO 8601 timestamp
   - `zone_id`: optional str
   - `dwell_ms`: optional int
   - `is_staff`: bool
   - `confidence`: float
   - `metadata`: optional JSON object

2. API validation model (`app/schemas/event.py`):
   - `event_id`: StrictStr
   - `store_id`: StrictStr
   - `camera_id`: StrictStr
   - `visitor_id`: StrictStr
   - `event_type`: one of:
     - `ENTRY`, `EXIT`, `ZONE_ENTER`, `ZONE_EXIT`, `ZONE_DWELL`, `BILLING_QUEUE_JOIN`, `BILLING_QUEUE_ABANDON`, `PURCHASE`, `REENTRY`
   - `timestamp`: timezone-aware datetime
   - `zone_id`: StrictStr | None
   - `dwell_ms`: StrictInt | None
   - `is_staff`: StrictBool
   - `confidence`: StrictFloat between `0.0` and `1.0`
   - `metadata`: dict[str, Any]
   - extra fields are forbidden

## Key Mismatches in `sample_eventsbe42122.jsonl`

### Field Name Mismatches

- `id_token` vs expected `event_id` or `visitor_id`
- `store_code` / `store_id` vs expected `store_id`
- `event_timestamp` / `event_time` vs expected `timestamp`
- `track_id` vs expected `visitor_id`
- `queue_event_id` vs expected `event_id`
- Root-level demographic fields (`gender_pred`, `age_pred`, `age_bucket`, `is_face_hidden`, etc.) are not captured under `metadata`
- `queue_join_ts`, `queue_served_ts`, `queue_exit_ts` are not recognized fields in the expected event schema

### Required / Optional Field Issues

- Sample events do not consistently provide `event_id`.
- Sample events do not use `visitor_id` consistently; zone and queue events use `track_id` instead.
- The expected schema requires `metadata` to be a JSON object; sample records do not include a `metadata` field at all.
- `is_staff` exists in some sample records, but not uniformly across event types.
- `confidence` is missing entirely from sample records.

### Data Type / Format Issues

- `timestamp` values in the sample are naive ISO strings like `2026-03-08T18:10:05.120000` with no timezone offset.
- Sample `store_id` values are strings in some records (`"ST1076"`) and `store_code` strings in others, while the pipeline generation code uses an integer store ID.
- The API validation schema uses `StrictStr` for `store_id`, which will reject numeric `store_id` values generated by `pipeline/run.py`.
- `event_type` values in the sample are lowercase or mixed-case (`entry`, `exit`, `zone_entered`, `zone_exited`, `queue_completed`) and are not in the allowed uppercase enum list.
- `is_revenue_zone` is a string `"Yes"` instead of a boolean or metadata object.

### Event Type / Semantics Issues

- Sample `event_type` values are not in the accepted set.
- `queue_completed` is not defined by the API or pipeline event model; the expected queue lifecycle event types are `BILLING_QUEUE_JOIN` and `BILLING_QUEUE_ABANDON`.
- `zone_entered` / `zone_exited` in the sample do not match expected `ZONE_ENTER` / `ZONE_EXIT`.

### Metadata Structure Issues

- The sample flattens demographic and queue details into root-level fields rather than embedding them inside `metadata`.
- The API schema has `extra = "forbid"`, so any additional root-level fields will be rejected unless they are moved into `metadata`.
- There is no standardized `metadata` object in the sample events.

## Submission Risk Assessment

The sample event schema would almost certainly fail ingestion under the current repository validation rules.

High-risk issues:

1. `event_type` values are invalid and do not match the allowed event type enum.
2. `timestamp` values lack timezone information and will fail validation.
3. Extra root-level fields are forbidden by `app/schemas/event.py` and will cause rejection.
4. Missing required fields such as `event_id`, `visitor_id`, `confidence`, and `metadata`.
5. `queue_completed` is an unsupported event type with non-standard queue field names.
6. The pipeline output model uses `store_id` as an integer, while the API model requires `StrictStr`, creating an additional mismatch for generated events.

## Recommended Fixes (Priority Order)

1. Align sample event payload field names with the API schema:
   - `event_id`, `store_id`, `camera_id`, `visitor_id`, `event_type`, `timestamp`, `zone_id`, `dwell_ms`, `is_staff`, `confidence`, `metadata`

2. Normalize event type values to the repository’s allowed list:
   - `ENTRY`, `EXIT`, `ZONE_ENTER`, `ZONE_EXIT`, `ZONE_DWELL`, `BILLING_QUEUE_JOIN`, `BILLING_QUEUE_ABANDON`, `PURCHASE`, `REENTRY`

3. Convert all timestamps to timezone-aware ISO 8601 format, e.g. `2026-03-08T18:10:05.120000+00:00`.

4. Consolidate extra event details into `metadata` instead of flattening them at the root.

5. Ensure every record includes a valid `event_id` and consistent `visitor_id` value.

6. Reconcile `store_id` typing across the pipeline and API:
   - Either emit `store_id` as a string consistently or change `app/schemas/event.py` to accept integer store IDs if numeric IDs are intended.

7. Replace unsupported queue event semantics with the expected queue lifecycle events and field names.

## Notes

- The repo uses `app/schemas/event.py` as the authoritative ingestion validation model.
- The sample file’s data shape appears to represent a different on-device / analytics event format than the API ingestion schema.
- Without an actual `output/` generated file to compare, the validation is based on inferred output structure from `pipeline/run.py` and the canonical schema in `app/schemas/event.py`.
